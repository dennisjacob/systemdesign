## Executive Overview

Understanding OS-level primitives is essential for system design interviews because every architectural decision ultimately manifests at the process, thread, and memory level. Interviewers probe your understanding of how processes communicate, how memory is managed, and how debugging works in production. This section covers the foundational knowledge that separates Senior from Staff+ engineers.

**Interview mindset**: OS primitives are not trivia. When you say "just add more threads," an interviewer expects you to know that threads share heap (race conditions), consume kernel memory (stack space), and require synchronization (lock contention). This section gives you the depth to defend those statements.

## N-Tier Architecture Patterns

### What It Is

The physical and logical separation of concerns into Presentation, Application, and Data tiers. In distributed systems, this expands to multiple layers of service boundaries, each representing a failure domain and a scaling unit.

### Core Dimensions

#### The Three-Tier Model

**Definition**: Presentation → Application → Data. Each arrow is a network hop, serialization point, and security boundary.

**Characteristics**:

- **Presentation Tier (Browser, Mobile App, API Gateway, CDN)**
  - **Cause**: Client devices have limited resources; need edge termination for security, SSL, rate limiting
  - **Approach**: API Gateway handles auth, rate limiting, request routing. CDN caches static/read-heavy content at edge.
  - **Trade-off**: Gateway adds ~1ms latency per request but protects downstream tiers from attacks and reduces load

- **Application Tier (Microservices, BFF, Auth, Business Logic)**
  - **Cause**: Business logic must be separated from data storage to allow independent scaling of compute vs. storage
  - **Approach**: Stateless instances behind load balancer, each handling a bounded set of responsibilities
  - **Trade-off**: More services = finer scaling granularity but N² possible communication paths

- **Data Tier (Relational DB, NoSQL, Cache, File Systems)**
  - **Cause**: Data persistence, consistency, and durability are fundamentally different concerns from request processing
  - **Approach**: Use the right storage for the right workload — OLTP (SQL), caching (Redis), analytics (columnar), files (S3)
  - **Trade-off**: Multiple storage types increase operational complexity but each optimizes for its workload

#### Architecture Type Comparison

**Definition**: How you organize the application tier.

**Characteristics**:

- **Monolith (e.g., early Shopify on Rails)**
  - **Cause**: Simplest deployment model — all code in one process, one deploy unit. Ideal when team and traffic are small.
  - **Approach**: Single app process → single DB. All modules communicate via in-process calls (μs latency).
  - **Trade-off**: Scales as one unit (no per-feature scaling). Any change redeploys everything. Best for <10 engineers.

- **3-Tier (e.g., traditional enterprise Java)**
  - **Cause**: Need to scale app tier independently from DB. Standard for moderate-size applications.
  - **Approach**: App servers behind LB → RDS with read replicas
  - **Trade-off**: All features share the same app tier bottleneck. Good for moderate scale, not hyperscale.

- **Microservices (e.g., Netflix)**
  - **Cause**: Independent teams need independent deploy cycles. Different services have different scaling needs (search scales differently than billing).
  - **Approach**: Each service owns its data store. API Gateway routes requests. Communication via gRPC/HTTP/messaging.
  - **Trade-off**: N services = N² possible communication paths. Distributed tracing required (Jaeger/Zipkin). Fault isolation is excellent but operational complexity is very high.

- **Modular Monolith**
  - **Cause**: Companies that tried microservices and scaled back needed the sweet spot — in-process speed of monolith with bounded-context discipline of microservices.
  - **Approach**: Strict internal module boundaries (enforced by package access modifiers or linting rules). Can extract to microservices later if needed.
  - **Trade-off**: Great balance of simplicity and structure. Used by Shopify, Stack Overflow.

**When to Use Each Architecture — Decision Framework**:

| Scenario | Best Choice | Why |
|----------|-------------|-----|
| **Startup (<5 engineers)** | Monolith | Fastest iteration, simplest deployment, one codebase |
| **Growing team (5-20 engineers)** | Modular Monolith | Scale code without operational complexity |
| **Large org (50+ engineers, multiple teams)** | Microservices | Independent deploys, team autonomy |
| **High-scale with uneven load** | Microservices | Scale search independently from checkout |
| **Legacy modernization** | Strangler Fig (hybrid) | Gradually extract services from monolith |
| **Strict latency requirements** | Monolith or Modular | Avoid network hops between components |

**Migration paths**:
```
Monolith → Modular Monolith → Selective Microservices → Full Microservices
                ↑ Most teams should stop here

Real examples:
- Shopify: Monolith → Modular Monolith (stopped, didn't go to microservices)
- Amazon: Monolith → Microservices (necessary at their scale)
- Segment: Microservices → Monolith (regretted microservices, consolidated)
```

### Trade-Off Matrix

| Pattern | Horizontal Scale | Fault Isolation | Latency | Operational Complexity |
|---------|-----------------|-----------------|---------|------------------------|
| **Monolith** | Poor (deploy together) | None (shared process) | Lowest (in-process) | Lowest |
| **3-Tier** | Moderate (app tier only) | Moderate | Medium (2 hops) | Moderate |
| **Microservices** | Excellent (per service) | Excellent | Highest (N hops) | Very High |
| **Modular Monolith** | Moderate | Good (bounded contexts) | Low | Low–Medium |

### Cross-Concept Connections

| Connection | Direction | Why |
|-----------|-----------|-----|
| **Tier boundaries → Stateless scaling (1.1)** | Cross-file | Stateless scaling happens within app tier; stateful data tier scales differently |
| **Microservices → Process isolation** | Within 1.2 | Each microservice typically runs as one or more OS processes |
| **Monolith → Thread model (below)** | Within 1.2 | Monolith uses threads; microservices use processes — different failure profiles |

### Failure Modes

- **Cascade failure across tiers**
  - **Cause**: App tier overload propagates to data tier — connection pool exhaustion → DB CPU spike → more timeouts → more retries → worse
  - **Approach**:
    - **Bulkheads**: Separate thread pools per downstream dependency
      ```
      Without bulkheads:                 With bulkheads:
      AppThread Pool (100 threads)      UserService pool: 30
      ├── UserService (slow)            PaymentService pool: 30
      ├── PaymentService                InventoryService pool: 40
      └── InventoryService              If UserService is slow,
      If UserService is slow,           only 30 threads affected
      ALL 100 threads wait on it        → Payment continues
      ```
    - **Circuit breakers** (Hystrix/Resilience4j): detect failure rate, open circuit, fail fast
    - **Rate limiting** at tier boundaries
  - **Trade-off**: Bulkheads reduce thread utilization; circuit breakers abandon requests that might succeed

- **Thundering herd at data tier**
  - **Cause**: Cache miss floods database tier simultaneously when many requests miss the same key
  - **Approach**: Request coalescing (one thread fetches, others wait), stale-while-revalidate, probabilistic early expiration
  - **Trade-off**: Coalescing adds ~1ms latency per miss but reduces DB load by N:1

## Process vs. Thread Mechanics

### What It Is

**Processes** provide memory isolation through separate virtual address spaces; **threads** within a process share memory. This single distinction explains almost every OS-level trade-off in systems design.

### Core Dimensions

#### Process Model

**Definition**: Independent virtual address space with its own stack, heap, file descriptors, page table.

**Characteristics**:

- **Memory isolation**
  - **Cause**: Separate page tables per process prevent one process from reading/writing another's memory. This is the OS's primary security and stability boundary.
  - **Approach**: Use processes when crash isolation matters (Nginx — one worker crash doesn't affect others), or when running untrusted code (Chrome — one tab crash doesn't crash the browser)
  - **Trade-off**: Isolation costs memory (duplicate page tables) and communication overhead (IPC required for data exchange)

- **Process creation cost (fork/exec)**
  - **Cause**: Fork must copy (or COW-map) the parent's page table; Exec must load a new binary, set up memory maps, resolve dynamic libraries
  - **Approach**: Pre-fork worker pools (Apache's `mpm_prefork`), container reuse (warm start instead of cold)
  - **Data**: Fork: 1–10ms (COW pages reduce actual copy cost); Exec: 10–50ms
  - **Trade-off**: Pre-forked processes consume memory even when idle

- **Context switch cost (10–100μs)**
  - **Cause**: TLB flush on every process switch. TLB maps virtual→physical addresses; a flush means the next N instructions will all TLB-miss (~100ns per miss) until TLB repopulates
  - **Approach**: Minimize process count per machine; use threads within a process for fine-grained work; use event loops to avoid context switches entirely
  - **Trade-off**: Fewer processes = less isolation

#### Thread Model

**Definition**: Multiple execution streams within the same process address space.

**Characteristics**:

- **Shared heap (race condition risk)**
  - **Cause**: All threads in a process share the same heap. Thread A reads a variable while Thread B writes it → non-deterministic result without synchronization
  - **Approach**: Use mutexes, read-write locks, atomics, or channels to synchronize shared state access
  - **Trade-off**: Synchronization adds latency (mutex lock ~25ns) and can cause contention cascades

- **Thread creation cost (0.1–1ms)**
  - **Cause**: Threads share the process address space — no page table setup needed, just allocate a stack (~8MB virtual, ~4KB committed initially) and a thread control block
  - **Approach**: Use thread pools to amortize creation cost across many tasks
  - **Trade-off**: Each thread's stack reserves 8MB virtual address space — 10K threads = 80GB virtual memory (though actual committed is much less)

- **Context switch cost (2–20μs)**
  - **Cause**: Threads share address space — partial TLB flush only (process-wide mappings stay cached). Still non-zero because register state and stack pointer change.
  - **Approach**: Match thread count to core count; use I/O multiplexing (epoll) to avoid thread-per-connection
  - **Trade-off**: More threads than cores → context switching overhead dominates

#### Why Context Switch Cost Matters at Scale

At 10K requests per second, even 10μs context switch overhead per request adds up:
```
10K RPS × 2 context switches/request × 10μs = 200ms of CPU time/second
= 20% of a single core wasted on scheduling overhead alone
```
**Cause**: Traditional thread-per-connection model pays this cost on every request. **Approach**: Event-loop architectures (Node.js, Nginx) eliminate context switches by handling many connections in one thread.

### Inter-Process Communication (IPC) Comparison

| Method | Latency | Throughput | Use Case |
|--------|---------|------------|---------|
| **Shared memory (mmap)** | ~100ns | Very high | Same-machine, high-frequency communication |
| **Unix domain socket** | ~5μs | High | Same-machine, different processes |
| **Pipe** | ~5μs | Moderate | Parent-child process, streaming |
| **TCP loopback** | ~50μs | Moderate | Same machine, network-oriented code |
| **Network socket** | 0.1–100ms | Lower | Cross-machine |

**Real example — Nginx worker model**: Nginx uses multiple worker processes (not threads) to avoid shared-state bugs. Workers communicate with the master via Unix domain sockets. Each worker handles thousands of connections with an event loop. This combines process isolation (crashes don't take down other workers) with event-loop efficiency.

### Trade-Off Matrix

| Aspect | Process | Thread |
|--------|---------|--------|
| **Memory isolation** | Strong (separate address space) | None (shared heap) |
| **Crash impact** | Isolated (only that process dies) | Whole process may crash |
| **Communication** | IPC overhead (pipe, socket, mmap) | Direct shared memory (fast but unsafe) |
| **Debugging** | Easier (separate core dumps) | Harder (interleaved stack traces) |
| **GC (JVM)** | Separate heap per process | Single heap — GC pauses affect all threads |
| **Python GIL** | Bypasses GIL (true parallelism) | GIL prevents parallel CPU execution |

**Process vs Thread — Detailed Decision Matrix**:

| Factor | Use Processes | Use Threads | Rationale |
|--------|---------------|-------------|-----------|
| **Crash isolation** | ✅ Critical (web servers, browsers) | ❌ Not needed | Chrome uses processes per tab; Nginx uses processes per worker |
| **Memory sharing** | ❌ Minimal sharing | ✅ Extensive sharing | Threads for in-memory cache; processes for isolation |
| **CPU-bound work (Python)** | ✅ Always | ❌ Never (GIL blocks) | Python multiprocessing bypasses GIL |
| **CPU-bound work (Java/C++)** | ❌ Threads are faster | ✅ Preferred | No GIL, shared memory faster than IPC |
| **I/O-bound work** | Either (async/event loop better) | Either (async/event loop better) | Modern approach: async I/O on few threads |
| **Security boundary** | ✅ Strong isolation | ❌ No isolation | Sandboxing untrusted code requires processes |
| **Scaling to 10K+ connections** | ❌ Too much overhead | ✅ If using thread pools | But event loops beat both |

**Real-world architecture examples**:

1. **Nginx (process-based)**:
   ```
   Master process (privileged, binds port 80)
     ├── Worker 1 (unprivileged, handles connections via epoll)
     ├── Worker 2
     └── Worker N (typically = CPU cores)
   
   Why processes?
     - Worker crash doesn't affect other workers
     - No shared memory = no race conditions
     - Each worker = independent event loop
   ```

2. **Apache MPM Prefork (process-based, legacy)**:
   ```
   One process per connection
   MaxRequestWorkers=256 → max 256 concurrent connections
   
   Why deprecated?
     - High memory (each process = 10-50MB)
     - Context switch overhead
     - Replaced by worker MPM (threads) and event MPM (async)
   ```

3. **Java application server (thread-based)**:
   ```
   Single JVM process
     ├── Thread pool (200 threads)
     ├── Shared heap
     └── Single GC affecting all threads
   
   Why threads?
     - Fast shared memory communication
     - JVM optimizations across threads
     - But: GC pause affects entire application
   ```

4. **Python CPU-bound (must use processes)**:
   ```python
   # Web scraper processing images
   from multiprocessing import Pool
   
   def process_image(img_path):
       # CPU-intensive: resize, filter, encode
       ...
   
   with Pool(8) as p:  # 8 processes, not threads
       p.map(process_image, image_paths)
   
   # Threads would be SLOWER due to GIL contention
   ```

**When to combine both**:
```
Master process
  ├── Worker process 1
  │     ├── Thread pool (for I/O)
  │     └── Event loop
  ├── Worker process 2
  └── Worker process N

Examples: Gunicorn with threads, Node.js cluster mode
```

### The Python GIL — A Critical Interview Point

```python
# Python threads do NOT run in parallel for CPU-bound work:
import threading
counter = 0
def increment():
    global counter
    for _ in range(1_000_000):
        counter += 1  # GIL acquired/released per bytecode instruction

t1 = threading.Thread(target=increment)
t2 = threading.Thread(target=increment)
t1.start(); t2.start(); t1.join(); t2.join()
# Result: counter < 2,000,000 (race condition despite GIL protecting bytecodes)
# AND slower than single-threaded for CPU work (GIL contention overhead)

# Solution: multiprocessing for CPU-bound work
from multiprocessing import Process  # separate OS processes, no GIL
```

**Cause**: Python's GIL ensures only one thread executes Python bytecode at a time, making threads useless for CPU-bound parallelism. **Approach**: Use `multiprocessing` (separate processes = separate GILs) for CPU work; use asyncio for I/O-bound work. **Connects to**: Concurrency models in 1.1 — Python can have concurrency (async I/O) but not CPU parallelism.

### Cross-Concept Connections

| Connection | Direction | Why |
|-----------|-----------|-----|
| **Context switch cost → Event loop (Coroutines)** | Within 1.2 | Event loops exist specifically to avoid context switch overhead |
| **IPC methods → DRAM/NUMA latency (1.3)** | Cross-file | mmap = DRAM ~100ns; Unix socket = μs-scale; network socket = ms-scale |
| **Python GIL → Concurrency section (1.1)** | Cross-file | GIL explains why Python needs event loops vs. threads for I/O |
| **Process isolation → N-tier boundaries** | Within 1.2 | Each microservice = separate process = crash isolation |

### Production Edge Cases

- **Fork bombs**: Uncontrolled process spawning (`:(){ :|:& };:`) exhausts PID space and memory. **Cause**: Recursive process creation without limit. **Approach**: `ulimit -u 256` per user, cgroups to enforce process limits in containers. **Trade-off**: Low limits break legitimate fork-heavy workloads (e.g., test runners).
- **Thread leaks**: Threads blocked on slow remote calls accumulate. At 1000 leaked threads, JVM heap fills with stack memory. **Cause**: Thread not returned to pool after task completes. **Approach**: Thread pool with bounded queue; audit thread creation in code review; set `uncaughtExceptionHandler` to log and cleanup. **Trade-off**: Bounded queues may reject requests during legitimate load spikes.
- **False sharing**: Two threads modify different variables on the same 64-byte cache line. Each modification invalidates the other core's cache line, causing 100× slowdown. **Cause**: CPU cache coherence protocol (MESI) invalidates the entire cache line on write. **Approach**: Pad variables to cache-line boundaries (64 bytes). **Connects to 1.3**: Cache line mechanics in volatile memory section.

## Coroutines and Async Runtimes

### What It Is

Coroutines enable cooperative multitasking without OS-level context switches. Essential for I/O-bound services handling thousands of connections on limited threads.

**The core insight**: If your thread spends 95% of its time waiting for I/O (network, disk), you need concurrency not parallelism. Coroutines give you thousands of concurrent I/O waits on a single thread — zero OS context switch overhead.

### Core Dimensions

#### Event Loop Model

**Definition**: A single thread manages many connections via I/O multiplexing (epoll/kqueue).

**Characteristics**:

- **Event loop mechanics**
  - **Cause**: Traditional thread-per-connection wastes memory and CPU (context switches, stack space). Event loops invert the model — one thread waits on many FDs and dispatches work when I/O is ready.
  - **Approach**:
    ```
    1. Register all FDs with epoll/kqueue
    2. epoll_wait() blocks until at least one FD is ready
    3. Dispatch coroutine/callback for each ready FD
    4. Coroutine processes data, issues new I/O request
    5. Register new FD interest → back to step 2
    ```
  - **Trade-off**: Any blocking operation in a coroutine stalls ALL connections handled by that thread

- **async/await vs. callbacks vs. promises**
  - **Cause**: Callbacks create deeply nested code ("callback hell"). Promises chain. async/await provides synchronous-looking syntax for asynchronous operations — all on the same event loop.
  - **Approach**:
    ```javascript
    // Sequential (await is a yield point):
    const a = await readFile('a.txt')
    const b = await readFile('b.txt')  // waits for a to complete first!
    
    // Parallel:
    const [a, b] = await Promise.all([readFile('a.txt'), readFile('b.txt')])
    ```
  - **Trade-off**: Sequential await is easy to read but slow; Promise.all is fast but any single rejection causes all to fail (use `allSettled` for fault tolerance)

#### Language Runtime Comparison

**Definition**: How different languages implement async execution.

| Runtime | Model | Concurrency Limit | Memory/Conn | CPU Parallelism |
|---------|-------|-------------------|-------------|----------------|
| **Node.js** | Single-threaded event loop | ~100K | ~2 KB | No (worker_threads for CPU work) |
| **Go** | M:N goroutine scheduler | ~1M goroutines | ~2 KB initial | Yes (GOMAXPROCS) |
| **Rust async** | State machines + Tokio executor | ~100K+ | ~1 KB | Yes |
| **Python asyncio** | Single-threaded event loop | ~10K | ~1 KB | No (GIL) |
| **Java Virtual Threads** | M:N on JVM (Project Loom) | ~1M+ | ~1 KB | Yes |
| **Erlang/Elixir** | Actor-based, own scheduler | ~10M+ | ~300 B | Yes |

**Cause**: Each runtime makes different trade-offs between memory per concurrent task, CPU parallelism support, and scheduler overhead. **Approach**: Match runtime to workload — Go/Java VTs for mixed workloads; Node.js/Python for I/O-heavy with light CPU; Erlang for massive concurrency with fault tolerance.

### Trade-Off Matrix

| Pattern | CPU Efficiency | Debugging Experience | Stack Traces | Best For |
|---------|--------------|---------------------|-------------|---------|
| **Thread pool** | Moderate | Easy (full stack traces) | Full | CPU-bound work |
| **Event loop** | Excellent | Hard (async frames) | Virtual | I/O-bound, high concurrency |
| **M:N threading (Go)** | Good | Medium (hybrid traces) | Hybrid | Mixed workloads |
| **Actor model** | Good | Moderate | Per-actor | Fault-tolerant distributed systems |

### Cross-Concept Connections

| Connection | Direction | Why |
|-----------|-----------|-----|
| **Event loop → Concurrency vs. Parallelism (1.1)** | Cross-file | Event loops implement concurrency (not parallelism); Amdahl's Law doesn't apply the same way |
| **Blocking in async → Cache miss (1.3)** | Cross-file | A cache miss in the event loop thread stalls all connections — worse than thread-per-connection |
| **Go goroutines → Thread mechanics** | Within 1.2 | Goroutines are lightweight threads on the M:N model — cheaper than OS threads, more flexible than event loops |
| **Java VTs → JVM GC pauses** | Within 1.2 | Virtual threads reduce per-connection cost but GC pauses still affect all VTs equally |

### Failure Modes

- **Blocking in async context**
  - **Cause**: A `time.sleep()` in a Python async function, or a synchronous DNS lookup in Node.js, stalls the **entire event loop** — every other connection experiences latency spike
  - **Approach**: In async contexts, only call async I/O. For CPU-heavy work, offload to a thread/process pool. Go exception: goroutines can block freely — Go runtime detects and reparks them without blocking others.
  - **Trade-off**: Offloading to thread pool adds IPC cost; Go's runtime handles this automatically but with slightly higher scheduler overhead

- **Unhandled promise rejection (Node.js)**
  - **Cause**: An async function throws but the rejection is not caught — in Node.js this previously crashed the process, now emits a warning
  - **Approach**: Always use `.catch()` or `try/catch` in `async` functions; use process-level `unhandledRejection` handler as last resort
  - **Trade-off**: Process-level handler may mask bugs in specific functions

- **Goroutine leak (Go)**
  - **Cause**: A goroutine waiting on a channel that is never written to will leak forever — allocating stack memory that is never freed
  - **Approach**: Use `context.Context` for cancellation:
    ```go
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()
    result := <-fetchWithContext(ctx, url)
    ```
  - **Trade-off**: Context-based cancellation requires all goroutines to accept/check context — legacy code may not support it

## Client Runtime Engine Mechanics

### What It Is

Browser environments (V8 for Chrome/Node, SpiderMonkey for Firefox) and server-side JS runtimes manage call stacks, garbage collection, and rendering pipelines. This knowledge matters for frontend performance interviews and any interview about JavaScript-at-scale.

### Core Dimensions

#### V8 Memory Model

**Definition**: JavaScript engine memory architecture — Young Generation (Scavenger GC), Old Generation (Mark-Sweep GC), and how objects move between them.

**Characteristics**:

- **Call Stack + Heap**
  - **Cause**: JavaScript uses a stack for function calls and a heap for objects. The stack is fast but limited (~10K–50K frames); the heap is large but GC-managed.
  - **Approach**: Keep stack depth shallow (avoid deep recursion); minimize heap allocations in hot paths
  - **Trade-off**: Stack overflow crashes the process; heap GC pauses freeze the event loop

- **GC pause limits**
  - **Cause**: Minor GC (Young Gen) pauses ~1–5ms; Major GC (Old Gen) pauses ~50–500ms. During GC, the event loop cannot process any events.
  - **Approach**: Keep working set small; avoid creating many short-lived objects (they stress Young Gen); use object pools for frequently allocated types
  - **Trade-off**: Object pools add code complexity but reduce GC pressure

- **V8 Optimization Tiers**
  - **Cause**: V8 compiles JavaScript in stages: Ignition (fast-to-start bytecode) → TurboFan (optimized machine code for hot functions). Changing an object's shape (adding/removing properties) causes deoptimization back to Ignition.
  - **Approach**: Keep object structures stable (consistent property order, types) in hot loops. Avoid `delete obj.prop` or dynamic property additions.
  - **Trade-off**: Strict object shapes reduce flexibility but enable 5–10× JIT performance gains

#### Event Loop Anatomy

**Definition**: How the browser event loop prioritizes tasks.

**Characteristics**:

- **Microtasks vs. Macrotasks**
  - **Cause**: Promises and `queueMicrotask` use microtasks (higher priority); `setTimeout`, `setInterval`, I/O callbacks use macrotasks (lower priority). Microtasks drain completely before the next macrotask.
  - **Approach**:
    ```
    Console output:
    setTimeout(() => console.log('A'), 0)   // macrotask
    Promise.resolve().then(() => console.log('B'))  // microtask
    console.log('C')                        // synchronous
    // Output: C, B, A
    ```
  - **Trade-off**: Long microtask chains can starve macrotasks (including rendering), causing UI jank

- **16ms render budget (60fps)**
  - **Cause**: The browser aims to render at 60fps, giving 16ms per frame for all JS execution + layout + paint + compositing
  - **Approach**: Use `requestAnimationFrame` for visual updates, `requestIdleCallback` for non-urgent work, Web Workers for CPU-heavy tasks
  - **Trade-off**: Chunking work across frames adds complexity but prevents UI freezing

### Cross-Concept Connections

| Connection | Direction | Why |
|-----------|-----------|-----|
| **GC pause → Latency waterfall (1.1)** | Cross-file | GC pause is a component in the latency waterfall — 50–500ms per major GC |
| **Event loop saturating → Context switch (1.2)** | Within 1.2 | Both are mechanisms to handle many tasks; event loop avoids context switches but can't parallelize |
| **V8 heap → DRAM/NVMe tiers (1.3)** | Cross-file | Heap fits in DRAM; evicted data hits disk — 1000× latency penalty |
| **Layout thrashing → NUMA (1.3)** | Cross-file | Both reduce effective throughput by accessing data inefficiently |

### Production Edge Cases

- **Memory leaks in closures**
  - **Cause**: A closure captures its surrounding scope. If that scope references a large object, it never gets GC'd as long as the closure is reachable.
    ```javascript
    // Leak: closure retains bigData forever
    function setup() {
      const bigData = new Array(1000000).fill('x')
      button.addEventListener('click', () => {
        console.log(bigData.length)  // keeps bigData alive
      })
    }
    // Fix: Pass only what you need
    ```
  - **Approach**: Only capture the specific values needed, not entire parent scope. Use weak references (WeakMap, WeakRef) for optional data.
  - **Trade-off**: WeakRef is non-deterministic — data may be GC'd at any time

- **Layout thrashing (forced synchronous layout)**
  - **Cause**: Reading a layout property (offsetWidth, scrollTop) after a style write forces the browser to recalculate layout *synchronously*, making 60fps animations impossible
  - **Approach**: Batch reads then writes:
    ```javascript
    const widths = elements.map(el => el.offsetWidth)   // batch reads
    elements.forEach((el, i) => el.style.width = widths[i] + 10 + 'px')  // batch writes
    ```
  - **Trade-off**: Batching reads and writes requires restructuring code — not always feasible with third-party components

- **Event loop saturation**
  - **Cause**: A synchronous for-loop over 10M items blocks UI for seconds because the event loop can't process renders or user input
  - **Approach**: `requestIdleCallback()`, `setTimeout(chunk, 0)` to yield between chunks, or move to a Web Worker
  - **Trade-off**: Chunking adds latency to the overall operation; Web Workers can't access the DOM

## Core Linux Resource Diagnostics

### What It Is

Production debugging starts with understanding system-level metrics. Interviewers value candidates who can diagnose "why is my service slow?" from first principles. This is also critical for SRE-track interviews.

### Core Dimensions

#### Essential Commands Reference

**Definition**: The Linux toolbox for diagnosing resource bottlenecks.

| Command | Purpose | Key Metric to Watch |
|---------|---------|---------------------|
| `vmstat 1` | Memory, CPU, swap | `si`/`so` (swap in/out > 0 = memory pressure) |
| `iostat -x 1` | Disk utilization | `%util` > 80% (disk saturated); `await` > 20ms (I/O slow) |
| `netstat -tulpn` | Network sockets | `TIME_WAIT` count (> 10K = port exhaustion risk) |
| `lsof -p PID` | Open file descriptors | Count > 80% of `ulimit -n` = FD leak |
| `pidstat -p PID 1` | Per-process CPU/mem | `%CPU` spike; `VSZ`/`RSS` growth (memory leak) |
| `ss -s` | Socket summary | CLOSE-WAIT accumulation = app not closing sockets |
| `perf top` | CPU flame graph | Which function consumes most CPU cycles |
| `strace -p PID -c` | System call profile | Which syscall is slowest; how often called |

**Cause**: Without these tools, debugging is guesswork. Each tool targets a specific resource layer. **Approach**: Start with `vmstat` to identify the bottleneck type, then drill down with the appropriate specialized tool.

#### Diagnosing "My Service Is Slow" — A Decision Tree

**Cause**: Service latency spiked. The root cause is unknown — could be CPU, memory, disk, network, or downstream dependency.

**Approach**: Systematic drill-down:

```
Service latency spiked → What changed?
│
├── CPU: vmstat → us% high?
│   ├── Yes: perf top → which function? → profile application
│   └── No: continue
│
├── Memory: vmstat → si/so > 0?
│   ├── Yes: swapping! → add RAM or reduce heap
│   └── No: check RSS growth with pidstat (memory leak?)
│
├── Disk: iostat -x → %util > 80%?
│   ├── Yes: I/O bound → add read replicas, add cache, NVMe upgrade
│   └── No: continue
│
├── Network: netstat → TIME_WAIT > 10K? CLOSE_WAIT accumulating?
│   ├── TIME_WAIT: set SO_REUSEADDR, tune tcp_tw_reuse
│   └── CLOSE_WAIT: application not closing sockets → code bug
│
└── Downstream: are external dependencies slow?
    └── Use distributed tracing (Jaeger/Zipkin) to find which service
```

#### eBPF — Modern Observability Without Performance Tax

**Cause**: Traditional tools like `strace` add 50–100% overhead by intercepting every syscall. eBPF programs run in-kernel with near-zero overhead because they aggregate in kernel space, not userspace.

**Approach**:
```bash
# Track read latency distribution (no code changes):
bpftrace -e '
  tracepoint:syscalls:sys_enter_read { @start[tid] = nsecs; }
  tracepoint:syscalls:sys_exit_read  /@start[tid]/ {
    @read_latency_ns = hist(nsecs - @start[tid]);
    delete(@start[tid]);
  }
'

# Count syscalls by process:
bpftrace -e 'tracepoint:raw_syscalls:sys_enter { @[comm] = count(); }'

# Track TCP connections by destination:
bpftrace -e 'kprobe:tcp_connect { printf("%s -> %s\n", comm, ntop(2, arg1->daddr)); }'
```

**Trade-off**: eBPF requires kernel 4.9+ and root/sudo access. Not all environments support it (containers may lack kernel access). **Why this matters in interviews**: Mentioning eBPF/bpftrace signals you understand production debugging at the kernel level without code instrumentation — a strong Staff+ signal.

### Trade-Off Matrix

| Tool | Overhead | Granularity | When to Use |
|------|----------|-------------|-------------|
| `top`/`htop` | <1% | Per-process aggregate | First look at CPU/memory |
| `vmstat`/`iostat` | <1% | System-wide | Identify resource bottleneck type |
| `strace -p PID` | 50–100% | Per-syscall, per-process | Dev/staging only — too slow for prod |
| `perf record` | 1–5% | CPU sampling | CPU profiling in prod |
| `eBPF/bpftrace` | <1% | Custom aggregations, kernel-level | Production observability |
| Application APM (Datadog, NewRelic) | 2–5% | Function-level traces | Full distributed tracing |

### Cross-Concept Connections

| Connection | Direction | Why |
|-----------|-----------|-----|
| **vmstat/iostat → SLO/SLI measurement (1.1)** | Cross-file | These tools provide raw SLI data for latency and availability |
| **iostat → SSD/HDD latency (1.3)** | Cross-file | `await` > 20ms signals HDD instead of SSD or saturated NVMe |
| **Memory decision tree → Fragmentation (1.3)** | Cross-file | Memory pressure can be fragmentation, not shortage |
| **eBPF → Production debugging** | Within 1.2 | Directly applicable to Interview Question #3 (memory leak) |

## Interview Questions

### ⭐ Process vs. Thread Selection

**Tests dimension**: Process vs. Thread Mechanics — isolation vs. performance trade-offs.

**When would you choose multiple processes over multiple threads for a web server?**

<details>
<summary>Solution Framework</summary>

**Choose multiple processes when**:
- **Language has a GIL** (Python, Ruby): Threads can't truly parallelize CPU work. Gunicorn spawns N worker processes to use N cores.
- **Crash isolation is critical**: A segfault in one process doesn't take down others. Nginx uses this — each worker handles thousands of connections; a crash kills only one worker.
- **Memory isolation needed**: Different tenants, security boundaries. Each process has its own address space.
- **Long-running jobs**: A leaking process can be killed and restarted independently.

**Choose multiple threads when**:
- **Shared data is large**: Sharing a 10GB in-memory index is free with threads; processes would require expensive IPC or shared mmap.
- **Low-latency IPC needed**: Threads communicate via shared memory (~100ns); processes need pipes/sockets (~5μs).
- **Language handles it** (Java, Go, C++): No GIL; threads truly parallelize.

**Real example**: Apache HTTPd offers `mpm_prefork` (processes) and `mpm_worker` (threads). PHP apps use prefork (thread safety issues); static content servers use worker (lower memory per connection).
</details>

### ⭐⭐⭐ Diagnosing a Production Memory Leak

**Tests dimension**: Linux diagnostics + V8 runtime mechanics.

**Your Node.js service is OOMing after 6 hours but restarts fine. Walk me through diagnosing and fixing it.**

<details>
<summary>Solution Framework</summary>

**Step 1 — Confirm it's a leak, not a spike**:
```bash
pidstat -p $(pgrep node) 10 | awk '{print $6, $7}'  # VSZ, RSS every 10s
# Or via Node process metrics:
setInterval(() => {
  const used = process.memoryUsage()
  console.log(`Heap: ${used.heapUsed/1e6}MB, RSS: ${used.rss/1e6}MB`)
}, 30_000)
```

**Step 2 — Take and compare heap snapshots**:
- Node `--inspect` → Chrome DevTools → Memory → Heap Snapshot
- Compare t=0, t=1h, t=2h — objects that grew between snapshots are the leak

**Step 3 — Common Node.js leak patterns**:

| Pattern | Symptoms | Fix |
|---------|----------|-----|
| Event listener not removed | Heap grows proportional to request count | `emitter.removeListener()` in cleanup |
| Closure over large objects | Heap grows proportional to closure count | Pass only needed values |
| Cache without eviction | Heap grows proportional to unique keys | LRU cache with max size |
| Unresolved Promises | Pending promise objects accumulate | Timeout + `.catch()` to all awaits |
| Global Map/Set | Map grows, never cleared | Bound size or use WeakMap |

**Step 4 — Quick fix while investigating**:
- HPA to restart OOMed pod automatically
- Limit heap: `node --max-old-space-size=2048` (OOM early, not gradual)
- Health check that fails if `process.memoryUsage().heapUsed > 1.5GB`

**Step 5 — eBPF to find allocations**:
```bash
bpftrace -e 'uprobe:/usr/bin/node:_Znwm { @alloc[ustack()] = sum(arg0); }'
```

**Key insight**: "Restart on OOM" is a bandaid, not a fix. Find the allocation site, fix the lifetime issue — usually 1 line of code (`.destroy()`, cap a Map size, use WeakRef).
</details>

## Quick Reference: OS Concepts Interview Cheat Sheet

| Concept | 1-Line Definition | When Interviewers Ask About It |
|---------|------------------|-------------------------------|
| **Virtual memory** | Each process sees its own 64-bit address space; OS maps to physical via page table | "How do you isolate processes?" |
| **TLB** | Cache of virtual→physical address mappings; flushed on context switch | "Why is process switch more expensive than thread switch?" |
| **Copy-on-Write (fork)** | Fork shares parent's pages until either writes; then copies | "How does fork() work efficiently?" |
| **epoll / kqueue** | OS mechanism to monitor thousands of FDs for I/O readiness | "How does Node.js handle 100K connections?" |
| **Page fault** | Access to unmapped memory; OS loads from disk or kills process | "What causes a sudden CPU spike in a memory-mapped file service?" |
| **OOM Killer** | Linux kills the process with highest OOM score when memory exhausted | "What happens when my container exceeds its memory limit?" |
| **cgroups** | Kernel feature to limit CPU, memory, I/O per process group | "How do containers enforce resource limits?" |
